# Release Instructions

Release notes are auto-generated by GitHub from merged PR titles and labels (see [`.github/release.yml`](.github/release.yml)). Every PR must carry one of: `bug`, `feature`, `internal`, `dependencies`, `change`, `release` — enforced by [`.github/workflows/pr-labels.yml`](.github/workflows/pr-labels.yml).

> 💡 **Tip:** The [`buildkite-agent-metrics-release` skill](.agents/skills/buildkite-agent-metrics-release/SKILL.md) walks an agent through the steps below. Use it instead of doing this manually.

1. Determine the SemVer version for the new release: `$VERSION`. Note: [semver does not include a `v` prefix](https://github.com/semver/semver/blob/master/semver.md#is-v123-a-semantic-version).
2. Create a new branch pointing to the trunk: `git fetch origin && git switch -c release/$VERSION origin/main`.
3. Update [`version/version.go`](version/version.go) with the new version.
4. Preview the auto-generated release notes (read-only API call, doesn't create a release):
   ```bash
   gh api -X POST repos/buildkite/buildkite-agent-metrics/releases/generate-notes \
     -f tag_name=v$VERSION \
     -f target_commitish=main \
     --jq .body
   ```
   If any PRs are mis-categorised (e.g. landed under 🏠 Internal because they had no labels), fix the labels on those PRs and re-run.
5. Push your branch, open a PR with title `release: v$VERSION`, body set to the preview notes from the previous step, and the `release` label applied. The `release` label both satisfies the label linter and excludes this PR from its own auto-generated notes.
6. Wait for CI to pass, then merge.

That's it. The [pipeline](https://buildkite.com/buildkite/buildkite-agent-metrics) build on main detects the version bump, creates the git tag via `gh release create`, and publishes the release with auto-generated notes.
